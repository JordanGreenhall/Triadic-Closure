# Item 2 load audit — `results/with-to-this-closure.md`

Question: is `with-to-this-closure.md` asking Item 2 / Born-rule structure only to assign/compare weights among realizable closure forks, or to prove forced collapse / attractor dynamics by itself?

Answer: it uses Item 2 for weighted actuality-selection / realizability-weighting. It does **not** use Item 2 alone to prove the unique attractor. The forcedness is carried by the local attractor computation and by the no-sub-arity-fork claim. Item 2 is load-bearing as the selection-weighting rule inside that dynamics, not as the proof of the attractor itself.

## Bottom line

`with-to-this-closure.md` is **not** using Item 2 to prove something the Born rule cannot prove by itself.

It is using Item 2 properly for the Born-rule-level job:

- given a realizability field / fork,
- selection is weighted by coherence-participation / Born-type measure.

The forced-attractor claim is then carried by additional local structure:

- complete-closure fork exists only at arity;
- singlet/closure fork is the unique stable attractor in the tested mutual-conditioning flow;
- non-closures drain into it;
- below arity there is no closure fork to actualize.

Therefore this page does not make Item 2 fail merely because it contains forced-collapse/attractor language. The audit target should be whether Item 2 supports the weighted-selection/readout rule it is actually asked to supply, and whether the local attractor computation is adequate on its own terms.
